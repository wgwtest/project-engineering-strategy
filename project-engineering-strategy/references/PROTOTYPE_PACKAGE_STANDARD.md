# Prototype Package Standard

本文件定义跨项目复用的原型与附图生成、归档、说明和评审规则。只要任务涉及页面原型、效果图、HTML 原型、截图比对、Canvas / 浮动布局原型或用户要先看原型再开发，就必须执行本规则。

## 1. 正式目录

默认正式原型根目录：

`DOC/CODEX_DOC/08_原型与附图/`

默认历史原型与评审记录目录：

`DOC/CODEX_DOC/07_过程文档/05_历史原型与评审记录/`

规则：

1. 当前仍可作为设计评审、前端实现和 UI 资产化依据的原型，放在 `08_原型与附图/`。
2. 未通过、被替代、只保留批注价值或不再作为当前实现依据的版本，迁入 `07_过程文档/05_历史原型与评审记录/`。
3. 如果既有项目已经使用 `03_规范与流程/04_原型与附图/` 或其他目录，必须在 `00-本地工程策略映射.md` 中显式声明该项目的正式原型根；未声明时按 `08_原型与附图/` 处理。
4. 不得把 `docs/superpowers/`、`.superpowers/`、`output/`、隔离 worktree 或临时目录中的图当作正式原型事实源，除非已经同步进正式原型根。

## 2. 版本包命名

每个原型方向、页面组、局部切片或重要修订，必须创建独立版本包目录：

`YYYY-MM-DD-HHMMSS-<项目或范围>-<原型主题>-vN/`

示例：

`2026-04-29-122600-CodeFactoryV2-P6门户扩展原型-v1/`

规则：

1. 一版一目录，不在同一目录中混放多个方向。
2. 新方向、新评审轮、新关键布局、新事实源变化，创建新版本目录。
3. 已经给用户评审、批注、通过或明确拒绝的版本不得原地覆盖。
4. 轻微错字、路径修正或 README 补证据可在同版修正，但必须不改变被评审图的设计结论。
5. 同一天多次生成必须使用精确到秒的时间戳，避免排序和引用混乱。

## 3. 版本包内容

每个版本包至少包含：

1. `README.md`
   - 本版唯一评审入口，必须图文混排。
2. `source/`
   - 可再生成原型的原始 HTML、CSS、JS、脚本、配置或其他可编辑源文件。
3. 渲染后的评审图片
   - 放在版本包根目录，使用可读文件名，例如 `01-桌面总览.png`、`02-节点选中态.png`、`03-移动端纵向观察面.png`。
   - 文件名应包含关键画板规格，例如 `1920x1080`、`390x844`，避免用非目标视口截图冒充目标比例原型。
4. `original/` 或 `source/original/`
   - 保存用户提供的原始图片、PDF、截图、手绘图、上游设计稿导出等不可再生成但影响本版判断的原始材料。
   - 如果本版没有外部原始图片，README 中明确写 `本版无外部原始图片`。
5. `REVIEW_RESPONSE.md`
   - 当本版来自上一版评审意见、批注处理或拒绝原因修正时必须提供。

可选内容：

1. `assets/`：本版专用图片、字体、图标、纹理等辅助资产。
2. `mapping.md`：当 README 已过长时，用于展开原型页到运行态路由、组件、数据源和验收方法的映射。
3. `scripts/`：截图生成、批量导出或像素检查脚本；如果脚本较少，也可放入 `source/`。

## 4. README 必备结构

`README.md` 必须让用户能直接批注每张图，并能理解每张图为什么这样设计。

推荐结构：

1. 标题与元信息
   - 原型名称、版本、生成时间、当前状态、适用 WBS / 页面、目标路由、所属应用、页面主对象、目标画板规格。
2. 本版定位
   - 本版解决什么问题、继承哪个版本、只处理哪些方向。
3. 非目标
   - 本版不解决什么，避免把原型误解为全面重做。
4. 共用事实源与设计依据
   - 列出需求、设计说明、接口规范、已有页面、用户批注、历史版本和数据模型等事实源。
5. 画板规格与布局预算
   - 写明桌面、移动端或其他目标画板的宽、高、比例、截图视口、固定区和可变区。
   - 对桌面整页原型，如无项目特殊约束，默认主画板为 `1920 x 1080` 的 `16:9` 画板。
   - 如果使用其他比例，必须写明原因，例如目标设备、嵌入容器、历史系统约束或用户要求。
6. 图文证据链
   - 每张图一个小节。
   - 每张图必须包含：评阅状态、画板规格、设计依据、需要用户判断的问题、允许偏差、图片嵌入。
7. 原始材料说明
   - 指明 `original/` 或 `source/original/` 中每个原始材料如何影响本版。
8. 原型到实现映射
   - 至少写明目标路由、页面归属、主对象、组件组、数据 / API 来源、验收方法或截图比对入口。
9. 允许偏差与不可接受偏差
   - 区分实现时可自然变化的样式细节与会阻塞通过的结构偏差。
10. 查看与再生成
   - 写出打开源文件和重新生成每张截图的命令。
11. 评审结论与后续处理
   - 标注 `待用户确认`、`通过`、`部分通过`、`不通过`、`已被后续版本替代` 等状态。

硬性要求：

1. 每张评审图都必须有设计依据；没有设计依据的图不能作为正式原型图。
2. README 不能只罗列图片，也不能只写“风格参考”“布局如图”。
3. 设计依据必须能追溯到正式文档、用户批注、上游原始材料、已确认原型或明确的实现约束。
4. 当设计依据属于推断，必须写成 `推断` 或 `设计判断`，不能伪装成已确认事实。
5. 原型图必须按目标画板比例设计，不得先做任意尺寸页面再随手截图。
6. 若目标是整页桌面界面，默认必须提供 `1920 x 1080` 的整体图；`1440 x 900`、浏览器当前窗口尺寸或缩放后的截图不能替代 `1920 x 1080` 整体图。

## 5. 原型生成流程

创建或刷新原型时按以下顺序执行：

1. 读取事实源
   - 最近交接、当前计划、设计说明、规范、验收入口、已有运行页面、用户批注和历史原型。
2. 确定版本包
   - 决定正式原型根、版本目录名、状态和本版范围。
3. 确定画板规格
   - 先声明目标宽高、比例、截图视口和布局预算。
   - 桌面整页原型默认使用 `1920 x 1080`；若不是这个规格，必须先写明理由。
   - 设计源文件应使用与目标截图一致的画板尺寸或等价的固定外框，不得依赖浏览器任意窗口撑开。
4. 编写或整理源文件
   - HTML / CSS / JS 原型放入 `source/`；外部原始图片放入 `original/` 或 `source/original/`。
5. 渲染评审图片
   - 截图视口必须等于 README 声明的目标画板规格。
   - 至少覆盖目标主视口；页面有响应式要求时同时提供桌面和移动端。
6. 截图自检
   - 必须实际查看生成截图，检查比例、遮挡、重叠、裁切、文字溢出和主要对象是否在安全区内。
   - 对固定面板、画布、安全区、底部栏等关键区域，应尽量用 DOM bounding box 或等价脚本检查是否出现非预期重叠。
   - `file` 显示图片尺寸正确只证明导出尺寸正确，不证明原型布局合格。
7. 写 README
   - 先写清设计依据、图文证据链和偏差规则，再请求用户评审。
8. 机测取证
   - 验证源文件可打开、图片文件存在且尺寸正确、README 无明显占位符、再生成命令可执行或已说明限制。
   - 记录截图自检结论；若发现重叠、裁切或比例不符，必须创建修正版，不能把该图送审。
9. 同步引用
   - 更新索引、计划、节点合同、验收方法、机测记录和交接中涉及的原型路径。

## 6. Canvas / 浮动布局原型规则

对 Canvas、节点图、浮动画布、拖拽布局、关系线或自由布局页面，传统线框图不足以表达交互时，版本包必须尽量补齐以下截图：

1. `总览态`
   - 表达整体空间、节点族群、关系线、密度和主要控制面。
2. `选中态 / 详情态`
   - 表达用户点选一个节点后，观察面、属性、输入输出、告警和动作入口如何出现。
3. `关系或路径态`
   - 表达当前链路、高亮路径、连线语义或底部关系轨。
4. `局部放大或小地图态`
   - 当画布密度高、审阅者难以看清时提供。
5. `移动端或窄屏降级态`
   - 如果页面需响应式展示，明确哪些内容保留、折叠或降级。

画布优先规则：

1. 如果用户定义的页面本体就是画布、图谱、星空式流程图或无边界空间，原型首图必须优先表现画布本身；固定面板、抽屉、实验柜、配置台和说明区不得常驻挤占首图主体。
2. 未选中状态应单独出图，表现整体空间、节点关系、数据流、状态氛围和主要观察方式。
3. 点击、选中、抽屉、弹层、悬浮工具、配置面板等交互状态应拆成单独状态图，不得把“最复杂堆叠态”当成唯一总览原型。
4. 如果某个功能本应是独立页面，例如配置实验柜、资产编辑器或管理工作台，不得为了展示完整性硬塞进门户主画布；应另建对应页面原型图。
5. 状态标签、模式、告警等轻量信息可以悬浮在画布边缘，但必须无背景或弱背景，不得破坏画布第一视觉。

README 中必须说明：

1. 哪些节点位置属于布局算法或用户拖拽结果，可允许实现偏差。
2. 哪些结构不可偏离，例如主画布必须存在、节点关系必须可辨认、数据源状态必须可见。
3. 哪些交互在静态图中只是状态表达，进入实现时需要真实事件链。
4. 整体画板中固定面板、浮层、抽屉、工具栏、底部轨和画布安全区的布局预算。
5. 如果右侧抽屉、左侧实验轨或底部关系轨会遮挡画布内容，应改为预留布局区域或拆成单独状态图，不得用互相压盖的总览图送审。

## 7. 评审与迭代规则

1. 用户尚未确认时，状态写 `待用户确认`，不得写成已批准实现标准。
2. 用户对某版提出批注后，先写 `REVIEW_RESPONSE.md` 判断每条批注是否成立、影响哪些图片、是否需要开新版本。
3. 如果需要改变被评审图片的设计结论，创建下一版目录，不覆盖旧版。
4. 已通过版本可留在 `08_原型与附图/` 作为实现事实源。
5. 已被替代或未通过版本迁入历史目录，并在当前目录 README 或索引中保留指向历史位置的简短记录。

## 8. 禁止事项

1. 禁止把散落 PNG、单个 HTML 或聊天截图当成正式原型包。
2. 禁止没有 README 设计依据就让用户评审。
3. 禁止把自动化测试通过、路由可达或运行页面已有外观当成原型评审通过。
4. 禁止在隔离 worktree 中生成正式原型后直接发给用户检查，必须同步到主交付目录或用户检查目录。
5. 禁止用后续实现截图倒推替换原型包，除非明确创建运行态截图比对记录。
