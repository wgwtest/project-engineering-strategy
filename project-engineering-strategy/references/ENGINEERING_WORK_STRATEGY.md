# Engineering Work Strategy

## 1. 目的

本文件用于固化跨项目复用的共享工程工作策略，作为多个项目后续开发、迭代、重构、验收和交接的统一执行基线。

目标有两个：

1. 为多个工程提供稳定、可重复执行的共享工作方法。
2. 为具体项目编写本地策略映射文档提供统一上位基线。

---

## 2. 核心原则

### 2.1 先理解，再修改

- 先读当前项目的设计文档、交接文档、计划文档和相关代码。
- 先确认目标边界、已有实现、未完成项和风险点，再开始改代码。
- 不在上下文不完整时直接做大范围改动。

### 2.2 先模型与契约，再实现与页面

- 先稳定共享模型、数据结构、接口契约和事件语义。
- 再修改后端服务与业务逻辑。
- 最后修改前端页面、展示层和交互层。

### 2.3 高内聚、低耦合

- 改动必须尽量集中在与当前目标直接相关的模块。
- 展示层、配置层、规则层、存储层、调度层应尽量解耦。
- 避免把短期需求直接写成跨模块硬编码。

### 2.4 小步迭代、持续可运行

- 每次只推进一个明确切片，不并行扩散多个高风险目标。
- 每完成一轮改动，都应保持系统可检查、可构建、可继续迭代。
- 不接受“先堆大量改动，最后一起修”的工作方式。

### 2.5 真实生效优先于表面完成

- 优先交付真正影响系统输出的改动，而不是只改文案、只改页面、只改占位字段。
- 配置项如果对外宣称“已支持”，则必须至少影响一个真实链路。
- 页面展示必须尽量基于运行态和正式数据，而不是本地猜测状态。

### 2.6 文档与代码同步

- 设计变更、边界调整、阶段结论、交接结果必须落文档。
- 文档不是补充说明，而是工程协作的一部分。
- 下一轮工作应能通过文档快速恢复上下文。
- 项目的正式文档根是默认用户审阅源；`docs/superpowers/` 与 `.superpowers/brainstorm/` 这类工作层文档不是默认用户审阅源。
- 因此，用户没有针对工作层文档逐条反馈，不得被解释为“用户已经看过但未反对”；只要其中内容已经足以影响正式设计、正式计划或正式实现，就必须同步进项目正式文档根。

### 2.7 语义优先于字面噪声

- 用户输入可能来自语音转录，允许存在错别字、断句错误、重复词或个别语义残缺。
- 代理应优先按上下文、当前工程状态和用户长期工作习惯做语义纠偏理解，而不是机械按字面逐词执行。
- 如果用户意图可由上下文稳定推断，则直接按推断后的意图推进，并在必要时简短说明采用了哪种纠偏理解。
- 只有在多种解释都会显著影响实施路径、且无法通过本地上下文消歧时，才回问用户。

### 2.8 远端协作面先判断后变更

- 当项目存在 GitHub Project、GitLab 看板或其他远端协作面时，先判断该协作面是否已被当前项目正式采用为 `Tracker` 或共享执行面。
- “能访问 / 能看到 / 技术上能创建”不等于“当前轮可以改”。
- 只要动作会改变远端协作面的结构或协作者读写方式，例如新增字段、状态项、视图、自动化或其他 `schema`，就必须先确认：
  - 当前项目是否允许使用该远端工具
  - 当前轮是否允许代理修改远端 `schema`
- 在未获得明确允许前，可以读取远端现状，但不得擅自创建、重命名、删除或重定义远端结构。
- 详细执行口径见 `3.5.5A 远端 Tracker 启用与 Schema 变更门槛`。

## 3. 标准工作流程

### 3.1 开始前

- 阅读最近一次会话交接文档。
- 阅读当前阶段计划、验收清单和相关设计文档。
- 检查工作区状态，识别已有未提交改动。
- 明确本轮目标、非目标和验证方式。

### 3.2 开发中

- 先定位最小切入点，再实施改动。
- 优先新增独立模块或独立文件，而不是向已有大文件持续堆积逻辑。
- 对复杂改动，先抽象层，再接业务层。
- 对已有用户改动、业务配置、构建产物保持谨慎，不随意回退。
- 默认直接在 `main` 分支推进、提交和推送；只有用户明确要求隔离分支，或当前任务需要高风险试验隔离时，才建立功能分支。

### 3.2A 正式工作目录与 Worktree 同步规则

为避免“改在隔离目录、用户看在主目录、双方看到的不是同一份内容”这类协作事故，统一增加以下硬规则：

#### 3.2A.1 三类目录语义

每个工程都应明确区分三类目录：

1. `仓库主目录`
   - 用户指定的正式仓库工作目录
   - 也是默认 `git status`、`git commit`、`git push` 所应基于的主工作树
2. `用户检查目录`
   - 用户实际打开页面、阅读文档、做批注、做人工验收时所看的目录
   - 如果项目代码根与仓库根不同，必须在本地策略映射中显式写明
3. `隔离 worktree`
   - 仅用于高风险试验、并行开发、冲突隔离或多代理并行
   - 只能作为内部临时工作区，不得默认视为正式交付目录

#### 3.2A.2 对用户输出路径规则

1. 面向用户的正式文件路径、验收路径、文档路径、页面入口路径，默认只允许引用：
   - `仓库主目录`
   - 或 `用户检查目录`
2. 不允许把 `.git/worktrees/...`、`.config/.../worktrees/...`、临时沙盒路径、代理工作目录直接当成正式成果路径发给用户。
3. 如果某份成果当前只存在于隔离 worktree 中，则该成果尚未进入“可交付 / 可验收 / 可检查”状态。

#### 3.2A.3 Worktree 使用前提

1. 只有在以下情况才允许建立隔离 worktree：
   - 用户明确要求隔离开发
   - 需要并行处理多个低共享状态任务
   - 当前任务具有较高试验性，直接在主工作树推进风险过高
2. 即使使用了 worktree，也必须预先约定：
   - 最终正式落点仍是用户指定的主目录
   - 用户检查、人工验收、正式文档更新、正式提交说明都以主目录为准

#### 3.2A.4 强制同步门槛

隔离 worktree 中的改动，最晚必须在以下任一时刻之前同步回主目录：

1. 准备让用户打开文件检查之前
2. 准备让用户进行人工验收之前
3. 准备向用户汇报“已完成 / 已修复 / 已可测试”之前
4. 准备启动本地服务供用户验证之前
5. 准备更新正式验收清单、自测报告、设计文档、计划文档、交接文档之前
6. 准备执行正式 `git commit` / `git push` 之前
7. 本轮会话结束之前，如果本轮已经形成用户可见成果

换句话说：

- `worktree` 可以承接“内部开发中间态”
- 但不能跨过“用户可见门”还停留在隔离目录里

#### 3.2A.5 同步频率规则

1. 默认按“一个可检查切片一次同步”执行，不跨多个切片长期积压。
2. 只要某个切片已经达到“可以给用户看、可以给用户测、可以写入正式文档”的程度，就必须在同一轮完成同步。
3. 不允许出现“先连续做很多轮 worktree 内修改，等用户要看时再想起来回主目录”的工作方式。

#### 3.2A.6 未同步成果的状态限制

如果改动仍只存在于隔离 worktree，则：

1. 不得称其为正式已交付成果
2. 不得把该路径作为用户批注入口
3. 不得据此更新共享协作面中的正式路径引用
4. 不得把“隔离目录中的状态”误报为“主目录已同步状态”

#### 3.2A.7 本地策略映射要求

每个项目的本地策略映射文档都必须明确写出：

1. `仓库主目录`
2. `用户检查目录`
3. `正式文档根`
4. 当前是否允许使用隔离 `worktree`
5. 如果允许，worktree 与主目录之间的同步门槛和正式落点

### 3.2B `superpowers / brainstorming` 与正式文档根同步规则

为避免“内容写进工作层了，但用户并不会去看，随后双方对是否已经确认产生误判”这类协作事故，统一增加以下硬规则。

#### 3.2B.1 工作层与正式层的职责边界

1. `docs/superpowers/` 与 `.superpowers/brainstorm/` 的主要职责是：
   - 方案推演
   - 中间草稿
   - 过程留痕
   - 局部计划拆解
2. 项目正式文档根才是：
   - 用户默认审阅入口
   - 正式设计约束入口
   - 后续实现、验收、交接和追溯的权威入口
3. 因此，工作层文档可以保留，但不能替代项目正式文档根承担“正式已确认基线”的职责。

#### 3.2B.2 用户审阅事实源规则

1. 默认情况下，用户只被认为审阅了项目正式文档根中的内容，而不是 `superpowers` 或 `brainstorming` 产物。
2. 如果某项关键设计只写在 `docs/superpowers/specs/`、`docs/superpowers/plans/` 或 `.superpowers/brainstorm/` 中，则不能默认认为用户已经看到。
3. 因此，不允许把“用户没有对工作层文档提出异议”解释为：
   - 用户已经确认
   - 用户已经阅读
   - 内容可以继续作为正式基线推进

#### 3.2B.3 强制同步触发条件

只要满足以下任一条件，就必须在同一轮把工作层内容同步进项目正式文档根：

1. `spec`、`plan` 或 `brainstorming` 中已经形成会影响后续开发的关键结论，例如：
   - 页面结构与关键交互
   - 对象模型、数据模型、状态模型
   - 输入输出契约、接口边界、回流规则
   - WBS 拆分、实施顺序、质量门与验收口径
   - 导出、交付、运行、监控等长期约束
2. 该内容已经来自用户讨论后的收敛结果，即使用户没有直接打开工作层文件。
3. 准备向用户表述以下任一结论之前：
   - 已设计
   - 已明确
   - 已规划
   - 可以开发
   - 按此执行
4. 准备以该内容为依据继续执行以下任一动作之前：
   - 编码实现
   - 自测与人工验收准备
   - 交接文档编写
   - 正式 `git commit`
   - 正式 `git push`

#### 3.2B.4 正式落点映射规则

同步时必须按内容语义进入正式文档根，而不是只在索引或映射表里挂一个引用。

最小映射要求如下：

1. 设计与交互规则：
   - 同步到 `02_设计说明/`
2. WBS、阶段计划、实施顺序：
   - 同步到 `04_研制计划/`
3. 自测、验收、交接、处理记录：
   - 同步到 `06_测试文档/` 与 `07_过程文档/`
4. 本地策略、协作规则、路径约束：
   - 同步到项目正式文档根中的策略映射或治理文档

#### 3.2B.5 未同步内容的状态限制

如果关键内容还只停留在 `superpowers` 或 `brainstorming` 工作层，则：

1. 只能被称为：
   - 草稿
   - 过程稿
   - 工作笔记
   - 内部推演材料
2. 不得被称为：
   - 正式设计
   - 正式计划
   - 已确认基线
   - 用户已审阅结论
3. 不得直接作为：
   - 用户批注入口
   - GitHub 协作面的正式引用路径
   - 后续实现“已获确认”的依据

#### 3.2B.6 同轮提交与交接门槛

1. 如果某轮已经在工作层写出了关键设计或关键计划，但尚未同步进项目正式文档根，则该轮不应被表述为“正式文档已更新完成”。
2. 在以下动作之前，必须先完成正式文档根同步：
   - 面向用户汇报本轮正式结论
   - 让用户按文档进行检查或确认
   - 编写正式自测报告、验收清单、交接记录
   - 执行正式 `git commit` / `git push`
3. 不允许出现“工作层已经写了很多，但等用户要看时才想起来同步正式文档根”的执行方式。

### 3.3 开发后

- 执行类型检查、构建或最小验证命令。
- 生成本轮对应的人工验收清单。
- 将本轮验收清单作为独立文档单独提交。
- 将该独立验收清单提交单独推送到远端。
- 告知用户对应验收清单的提交号，便于远端审阅。
- 复核改动范围是否扩散到无关模块。
- 将验收清单路径和建议验收顺序明确告知用户。
- 等待用户基于验收清单给出明确验收结果。
- 记录本轮完成项、未完成项、风险项和下一步建议。
- 形成交接文档，确保下一轮可直接续做。
- 在用户未明确反馈验收结果前，本轮开发只算“已完成实现与待验收”，不算“阶段闭环”。

### 3.4 收到验收意见后

- 读取最新验收清单中的用户批注、总结和结论。
- 对人工验收清单，默认只把用户写在 `【】` 内的内容视为正式用户输入。
- 不要求用户主动修改由开发者维护的字段，例如：
  - `状态`
  - `验收结论`
  - `通过/未通过` 占位项
- 因此，开发者不得反向把自己写的文档字段当成用户验收依据；如果字段状态与 `【】` 内容冲突，应以 `【】` 内容为准。
- 人工验收相关的聚合页、索引页、导航页、README、Issue 状态、Project 状态都不是验收事实源；这些都只能由活跃验收清单中的用户 `【】` 批注派生同步，不得反向作为判断依据。
- 对用户以显式标记形式给出的意见进行逐条提取、归类和分析。
- 单独形成“验收意见处理文档”，记录问题摘要、原因判断、影响范围、候选处理方案和建议优先级。
- 先将该处理文档提交给用户审定，不直接跳过分析环节进入改计划或改代码。
- 如果验收意见已经明确指出 WBS 结构、验收口径或阶段拆分本身有误，优先重排计划与验收模型，不要先头痛医头地改叶子代码。
- 用户人工验收清单只允许出现用户可观察动作和文档核验动作，不允许把读代码、跑 `npm test`、跑 `npm build` 之类开发者自测动作塞给用户。
- 命令执行、单元测试、构建结果、类型检查、接口验证统一进入自测报告或执行契约，不直接写进人工验收清单。
- 只有在用户确认处理意见后，才进入下一步处置，例如：
  - 修改研发计划
  - 调整验收清单
  - 修订页面结构
  - 修订代码实现
- 对“审定意见处理”本身形成的文档更新，应自动单独提交并推送到远端，便于远端同步审阅。

### 3.6 GitHub 评论来源处置规则

当通过 GitHub Issue/PR 拉取评论并准备据此修改时，必须先做来源分级：

1. `Collaborator`（或项目 Owner/Maintainer）评论：
   - 可作为直接修改依据，按正常评审处置流程执行。
2. 非 `Collaborator` 评论（如 `CONTRIBUTOR`、`NONE`、外部账号等）：
   - 只能作为“候选意见”记录，不得直接触发计划或代码改动。
   - 必须先向用户发起确认，只有用户明确认可后，才可进入修改执行。

执行要求：

1. 在处理记录、issue 回复或交接文档中，标注评论来源级别与确认结果。
2. 若用户未确认，则状态保持“待确认”，不推进对应实现改动。

### 3.5 计划与协同管理策略

对于存在多阶段交付、多人协作、并行开发或用户验收节奏滞后的工程，默认采用三层协同管理模型：

1. `Tracker` 层
   - 推荐使用 GitHub Project 或同类看板
   - 负责排程、状态、时间、路线图和树形结构展示
2. `Execution Contract` 层
   - 推荐使用 GitHub Issue 或同类任务单
   - 负责单个任务的最小执行契约
3. `Local Docs` 层
   - 使用项目本地文档目录
   - 负责长篇设计说明、详细验收脚本、自测报告、交接记录和过程归档

#### 3.5.1 三层职责分工

- `Tracker` 只负责人类快速查看的计划与状态，不承载长篇细节。
- `Execution Contract` 必须做到“不看聊天记录也能开工，不翻代码也能找到验收入口”。
- `Local Docs` 负责承接 issue 不适合直接塞进去的长说明与证据。

推荐最小分工如下：

1. `Tracker`
   - `Status`
   - `Phase`
   - `Contributes To`
   - `Work Type`
   - `Start Date`
   - `Target Date`
2. `Execution Contract`
   - `工作目标`
   - `工作内容`
   - `潜在难点`
   - `Depends On`
   - `成果物`
   - `验收方法`
   - `当前状态`
   - `上级节点`
   - 按需扩展：`Owner / Write Scope / 证据链接`
   - 对数据规范、文件规范、导入导出类任务，`成果物` 还应额外点名 1 个小型但字段覆盖尽量全面的样例数据文件
3. `Local Docs`
   - 文档根：`README.md`、`00-本地工程策略映射.md`
   - `01_需求分析/`
   - `02_设计说明/`
   - `03_规范与流程/`
   - `04_研制计划/`
   - `05_节点合同/`
   - `06_测试文档/`
   - `07_过程文档/`

#### 3.5.1A 文档管理依据与标准边界

全局工程策略采用的是“标准兼容型本地研发文种体系”，不是对任一国际标准或军标目录的逐项照抄。

统一依据如下：

1. 生命周期与过程控制，主要吸收：
   - `ISO/IEC/IEEE 12207`
   - `ISO/IEC/IEEE 15289`
2. 计划与项目管理控制，主要吸收：
   - `ISO/IEC/IEEE 16326`
3. 需求与设计信息项组织，主要吸收：
   - `ISO/IEC/IEEE 29148`
   - `IEEE 1016`
4. 测试文档与验证证据组织，主要吸收：
   - `ISO/IEC/IEEE 29119`

上述标准在本策略中的作用，是提供工程控制点和信息项组织原则，而不是强制每个项目把本地正式文档根改写成标准原版目录。

因此，统一边界应写清为：

1. 项目本地正式文档根负责承接项目自身研发层的正式文档管理。
2. 如果项目后续需要面向客户、行业规范或军标交付形成标准化文种包，可在本地正式文档根之上再建立“标准投影层”或“交付投影层”。
3. 不允许为了追求“看起来像标准目录”，反向破坏本地研发协作所需的执行、验收、交接与追溯结构。

本策略允许存在本地扩展文种，但必须满足两个条件：

1. 该文种承担标准原版目录中没有被直接承接、但本地协作又稳定需要的职责。
2. 该文种的职责边界必须在项目正式文档中写清，不能靠聊天约定长期维持。

典型允许的本地扩展包括：

1. `05_节点合同/`
   - 用于承接单节点执行约定
   - 它不是标准原生命名，但在 `WBS -> 执行 -> 验收` 之间承担必要桥接责任
2. `06_测试文档/02_验收大纲/`
   - 用于承接稳定验收基线
3. `06_测试文档/05_验收清单/`
   - 用于承接面向人工验收的活跃操作单
4. `07_过程文档/`
   - 用于承接交接、历史计划和过程留痕

每个项目的 `00-本地工程策略映射.md` 必须显式写出：

1. 当前文档管理依据采用哪组标准控制点
2. 本地正式文档根与标准投影层之间的关系
3. 当前项目启用了哪些本地扩展文种，以及这些文种分别解决什么职责缺口

#### 3.5.2 WBS 树形拆分规则

- 计划应优先按 WBS 方式进行严格树形拆分。
- `L0` 表示总纲，`L1` 表示阶段，后续层级继续向下拆分为工作包和执行任务。
- 稳定计划文档按 WBS 节点命名，不使用“当前阶段计划”作为稳定文件名。
- `04_研制计划/` 默认一节点一文档，文件名统一使用 `NN-WBS-节点编号-主题-研制计划.md`。
- `02_设计说明/` 下的稳定设计正文默认使用 `WBS-节点编号-主题.md`，不再用日期前缀表达先后顺序。
- 无法直接归属具体节点的总设计稿，也应优先挂到最贴近的 `WBS` 编号下，避免同层出现日期命名与 `WBS` 命名混用。
- `03_规范与流程/04_原型与附图/` 作为静态原型稿、设计导出图和原型说明的固定落盘目录。
- 当前激活节点应记录在 README、索引文档和 `Tracker` 中，而不是通过改文件名表达。
- `M1 / M2 / M3` 这类里程碑是验收与验证节点，不应替代 WBS 结构成为顶层分解轴。
- WBS 标题必须优先暴露“对象边界”，而不是优先使用抽象动作词、流程词或模糊概括词。
- 如果一个节点同时覆盖两个并列对象域，例如“关卡 / 地图”、“技能 / Buff”，父节点标题必须直接点名这两个对象域，不允许把对象边界藏进正文。
- 评审者应能只看树标题，就大致判断当前拆分是否已经覆盖主要需求；如果看标题仍无法判断覆盖性，说明命名过于抽象，必须重写。
- 命名首先要回答“这棵子树到底在管理什么对象”，其次才回答“对这些对象做什么”；不允许先写抽象动作、再把真实对象藏到正文里。
- 若某父节点下存在多个并列对象域，则该父节点必须直接写成并列对象名，例如 `关卡与地图编辑`，而不是 `内容扩充`、`管理链`、`能力链` 之类无法审计覆盖面的标题。
- 编号层级必须与父子树严格一致；如果 `3.2.1` 是父节点，则它的直接子节点只能是 `3.2.1.1 / 3.2.1.2 ...`，不能把 `3.2.2 / 3.2.3` 挂在 `3.2.1` 下面。
- 不允许出现“树关系是父子，但编号关系像并列”的情况；一旦出现，优先重编编号，不保留暧昧折中写法。
- 当某个主树同时包含“设计对象”和“对应工具”两类工作时，优先按“对象设计 -> 对应编辑器/实现工具”拆分，不要把对象设计、工具实现和验证页平铺成一串同层叶子。
- 如果某个验证页、本样页、效果图页只是某个正式工具的前置验证，它应挂在该工具节点之下，作为子叶子存在，而不是长期平铺在总树根下。
- 同层兄弟节点必须使用同一拆分轴；例如不能一边按“对象域”命名，另一边按“执行动作”或“验证方式”命名。
- 同层兄弟节点的拆分轴必须能从标题本身看出来；如果一组兄弟节点里既有“元数据设计”，又有“编辑器”，又有“验证页”，则必须先补一个上层父节点，把它们重新分组后再挂树。
- 对标题进行自检时，至少回答三个问题：
  - 只看标题，是否能判断本层对象边界是否完整
  - 只看标题，是否能判断兄弟节点之间是否存在漏项
  - 只看标题，是否能判断验证页/样本页属于哪个正式对象域
- 典型正确示例：
  - `WBS-3.2 关卡与地图编辑`
  - `WBS-3.2.1 关卡地图元数据设计`
  - `WBS-3.2.4 关卡编辑器基础能力与 JSON 导出链`
  - `WBS-3.2.5 地图编辑器基础能力与地图包导出链`
- 典型错误示例：
  - `WBS-3.2 内容扩充`
  - `WBS-3.2 关卡管理与关卡编辑链`
  - 在同层同时放 `关卡元数据设计 / 地图编辑器 / 效果图验证页` 而不建立对象域父节点
- 对强交互多视图编辑器，先单独建立“共性机制 / 共性方案”父节点，再建立统一的“视图设计”父节点后进入各执行分支，不允许把多个强交互视图直接平铺成顶层节点。
- 对强交互多视图编辑器中的单个视图，默认至少拆成：
  - 要素设计
  - 能力验证
  - 功能验证
  - 样式验证
- 不允许继续使用“一个大章节 + 一个叶子节点”来承载复杂工作台。

#### 3.5.2A WBS 远端优先读取规则

为避免本地镜像滞后导致读取到过期树结构，WBS 的读取和判断统一增加以下规则：

1. GitHub `Issues + sub-issues` 是 WBS 树的唯一事实源。
2. GitHub `Project` 只作为排程与展示投影，不替代 Issue 树成为结构事实源。
3. 默认不再创建或维护本地 `issue tree mirror` 作为“当前树状态”的读取入口。
4. 当需要确认以下任一信息时，必须优先直接查询远端 GitHub：
   - 当前 WBS 树结构
   - 当前节点标题
   - 当前节点正文契约
   - 当前父子挂接关系
   - 当前远端状态字段
5. 如果远端暂时不可访问，只允许回退到仓库主工作树中的正式文档根作为本地快照参考：
   - 优先读取仓库主工作树 / 用户检查目录
   - 不使用隔离 `worktree` 中的本地镜像或派生文件反推当前远端树状态
6. 这种本地回退只代表“最后同步到主工作树的正式快照”，不代表远端当前事实；使用时必须明确标注为：
   - `本地快照`
   - 或 `待远端复核`
7. 已存在的 `issue tree mirror` 文件，如因历史原因保留，只能视为：
   - 历史快照
   - 过程证据
   - 辅助说明
   不能再作为当前 WBS 事实入口。

#### 3.5.3 双门模型

默认将项目推进状态分成两个门：

1. `研发推进门`
2. `阶段关闭门`

规则如下：

- `研发推进门` 决定后续工作是否可以继续推进。
- `阶段关闭门` 决定当前阶段是否可以正式标记为完成或关闭。
- 用户人工验收是阶段关闭的唯一依据。
- 用户人工验收不应默认阻断低耦合的后续准备工作。

#### 3.5.4 Prep / Core 规则

后续任务默认区分为：

1. `Prep`
   - 低耦合准备工作
   - 例如：契约草案、测试脚手架、样本数据、独立壳层、验收基线、只读分析
2. `Core`
   - 直接依赖上一阶段稳定输出的正式实现
   - 例如：绑定正式运行时、依赖稳定数据结构的业务逻辑、依赖上一阶段结论的主链功能

当上一阶段处于 `待人工验收` 时：

- 可以继续推进下一阶段的 `Prep`
- 不应把下一阶段 `Core` 视为正式主线开始

当上一阶段人工验收通过后：

- 才允许把相关 `Core` 任务作为正式主线推进

#### 3.5.5 状态语义分层

- `Tracker` 可继续使用粗粒度状态，便于路线图和看板阅读。
- `Execution Contract` 应使用更细的执行状态，例如：
  - `待开发`
  - `开发中`
  - `已自测`
  - `待人工验收`
  - `已验收`
  - `阻塞中`

这样做的目的，是把“排程视图”和“执行视图”解耦，避免一个字段同时承担两种责任。

如果 `Tracker` 使用 GitHub Project 且 `Status` 为单选字段，默认推荐使用以下粗粒度映射：

- `Todo`
  - 尚未正式进入执行
  - 或虽然已拆分，但当前轮未启动
- `In Progress`
  - 正在实现
  - 或正在根据验收意见返修
  - 此时不应误解为已经具备人工验收条件
- `Hum Check`
  - 已实现
  - 已完成本轮自测
  - 已提供人工验收入口或待重测入口
  - 当前正在等待用户人工核验、人工复测或人工确认
- `Done`
  - 已收到用户明确 `【通过】`、`【已通过】` 或同义显式结论
  - 并已同步本地验收清单、Execution Contract 与 Tracker

如果项目尚未设置 `Hum Check`，可临时把“待人工验收”映射为 `In Progress`；但一旦 Project 已提供 `Hum Check`，应优先使用，不再把待人工核验节点继续混放在 `Todo` 或普通 `In Progress` 中。

#### 3.5.5A 远端 Tracker 启用与 Schema 变更门槛

- 当项目存在 GitHub Project、GitLab 看板或其他可写远端 `tracker` 时，代理首先要判断：该协作面是否已经被当前项目正式采用为 `Tracker` 层，而不是把“能看到某个 board”直接当成可写事实。
- 如果项目尚未明确采用远端 `tracker`，默认不擅自创建字段、状态选项、视图或自动化；继续以本地 WBS 计划、执行合同和正式文档根承接状态与排程信息。
- 即使项目已经采用远端 `tracker`，也必须先确认当前协作权限与用户授权是否允许代理修改远端 `schema`。
- 这里的 `schema` 包括但不限于：
  - 字段
  - 单选状态项
  - 分组规则
  - 视图
  - 自动化
  - 其他会影响协作者读写方式的结构
- 在未获得明确允许前，可以读取远端现有字段和状态，但不得擅自新增、重命名、删除或重定义远端字段与状态项。
- 如果用户明确允许创建或调整远端 `schema`，默认推荐采用“双层状态模型”：
  - `Status` 保持粗粒度排程状态，例如 `Todo / In Progress / Hum Check / Done`
  - 另增 `Execution Status`、`当前状态` 或等价字段，承接：
    - `待开发`
    - `开发中`
    - `已自测`
    - `待人工验收`
    - `已验收`
    - `阻塞中`
- 只有在以下情况之一成立时，才建议把上述六态直接放进远端唯一 `Status` 字段：
  - 用户明确要求使用单字段模型
  - 当前工具无法稳定支持第二个状态字段
  - 现有项目协作面已经按该单字段模型运作，继续沿用的迁移成本明显更低
- 如果采用单字段模型，必须在同一轮显式说明其代价：路线图/排程视图与执行细状态会被耦合，看板可读性与跨项目一致性会下降。
- 对远端 `tracker schema` 的任何新增或调整，都应在同一轮同步到本地策略映射、启动模板或协作说明，避免远端模型已变化而本地执行口径仍停留在旧版本。

#### 3.5.6 时间估计与路线图

- 对于进入执行队列的工作项，应给出明确的开始时间与目标时间。
- 时间基线应优先同步到 `Tracker` 的 Roadmap 视图，而不是只写在本地文档里。
- 本地计划文档保留时间说明，是为了补充推导依据和当前激活节点解释，不替代路线图本身。

#### 3.5.7 证据回链规则

- 当任务进入 `已自测` 或 `待人工验收` 时，`Execution Contract` 应回链到对应：
  - 自测报告
  - 验收清单
  - 会话交接或交付记录
- 证据应尽量放在本地文档层维护，`Execution Contract` 只保留最小必要链接。
- 不要求把长篇验收脚本直接塞进 `Tracker` 或 issue 正文。

#### 3.5.10 人工验收与自测分层规则

- `06_测试文档/05_验收清单/` 面向用户人工验收，只保留：
- `06_测试文档/02_验收大纲/` 面向稳定验收基线，只保留：
  - 节点级长期通过条件
  - 验收入口、判据与通过门
- `06_测试文档/05_验收清单/` 面向用户人工验收，只保留：
  - 用户可观察交互
  - 页面可见结果
  - 文档存在性与条目核验
- `06_测试文档/01_自测报告/` 面向开发者验证，承接：
  - `npm test`
  - `npm build`
  - `npm typecheck`
  - 接口级验证
  - 自动化测试结果
- 如果人工验收清单里出现“读代码确认函数存在”“手工执行 npm 命令”之类条目，应视为验收设计错误并改写。

#### 3.5.10A 集中验收总入口生成规则

如果项目使用“当前待验收功能总入口”“今日待验收入口”“集中验收入口”这类总导航页，则该页默认属于正式验收资产，必须按稳定流程生成，不允许随轮次随意改口径。

统一要求如下：

1. 总入口必须放在项目正式文档根或正式验收目录中，不得放在临时目录、隔离 `worktree`、工具缓存目录或过程沙盒目录中。
2. 总入口必须使用固定章节顺序，默认至少包含：
   - `最后整理时间`
   - `状态`
   - `使用方式`
   - `统一前置准备`
   - `今日建议测试顺序`
   - `当前结论`
   - `补充说明`
3. 总入口中的 `最后整理时间` 必须在每次重新生成或刷新时同步更新，不能沿用旧时间。
4. 总入口中的 `当前结论` 只允许列出：
   - `待验收`
   - 或 `未通过待重测`
   的叶子节点。
5. 已通过节点不得继续出现在总入口正文、说明段落、留痕段落或“本轮移出列表”中；历史通过记录应移到目录索引或历史索引页。
6. 总入口中的建议测试顺序应按功能链或用户验收路径分组，不按文件名排序，不按提交时间机械排序。
7. 总入口必须与活跃验收清单目录和本轮同步后的状态一致；节点数量、节点编号和链接不得与目录现实状态脱节。

#### 3.5.10B 总入口刷新步骤

当需要生成或刷新集中验收总入口时，默认按以下步骤执行：

1. 先确认正式工作目录和用户检查目录，禁止在隔离 `worktree` 中直接生成面向用户的正式入口。
2. 扫描活跃验收清单目录，收集当前仍处于 `待验收` 或 `待重测` 的 WBS 叶子节点。
3. 读取本轮已经进入处理回路的用户 `【】` 批注；如果用户已明确给出 `【通过】` 或同义结论，应先完成状态同步，再重新生成总入口。
4. `总入口`、`README`、Issue、Project 都只是派生协作面，不是状态源；禁止根据这些聚合面反推节点状态。
5. 如果活跃验收清单与任一聚合协作面冲突，应一律以活跃验收清单中的用户 `【】` 批注为准，然后自上而下重算其他页面。
6. 更新总入口的 `最后整理时间`。
7. 回写稳定的 `使用方式` 和 `统一前置准备`，确保入口说明与当前工程的真实测试方式一致。
8. 根据当前工程的主流程、作者工具链、probe 页、回归页等真实使用关系，生成 `今日建议测试顺序`。
9. 回写 `当前结论`，确保只保留仍待验收的节点及其正式清单链接，不写入任何“已通过节点回顾”或历史留痕段。
10. 若当前没有任何待验收节点，应明确写出“当前无待验收节点”，而不是保留旧列表。
11. 刷新总入口后，应同步检查同目录的 `README` 或索引页是否需要更新，避免“总入口已变、索引仍旧”的状态漂移。

#### 3.5.10B.1 术语定义与判定伪代码

为避免“活跃清单”“读取批注”“状态同步”这类词变成模糊话术，统一固定为以下可执行定义。

固定目录：

- 验收清单正式目录：`DOC/CODEX_DOC/06_测试文档/05_验收清单/`
- 验收清单归档目录：`DOC/CODEX_DOC/06_测试文档/05_验收清单/归档/`
- 当前待验收总入口：`DOC/CODEX_DOC/06_测试文档/05_验收清单/00-当前待验收功能总入口.md`
- 验收清单目录索引：`DOC/CODEX_DOC/06_测试文档/05_验收清单/README.md`

术语定义：

1. `候选清单文件`
   - 指 `DOC/CODEX_DOC/06_测试文档/05_验收清单/` 目录下
   - 文件名匹配 `*人工验收清单.md`
   - 且不是 `00-当前待验收功能总入口.md`
   - 且不是 `README.md`
2. `活跃清单文件`
   - 指候选清单文件中，仍代表该 WBS 节点当前正式人工验收入口的那一份
   - 默认在正式目录，不在 `归档/`
3. `状态事实源`
   - 只指活跃清单文件中的用户 `【】` 批注
   - 不指文件名中的 `[待验收] / [已通过]`
   - 不指清单正文中的 `状态：`
   - 不指总入口、README、Issue、Project
4. `派生协作面`
   - 指文件名状态
   - `00-当前待验收功能总入口.md`
   - `README.md`
   - GitHub Issue 正文中的 `当前状态`
   - GitHub Project `Status`

状态判定伪代码：

```text
INPUT:
  checklist_dir = "DOC/CODEX_DOC/06_测试文档/05_验收清单/"

candidate_files =
  all files in checklist_dir
  where name endswith "人工验收清单.md"
  and name != "00-当前待验收功能总入口.md"
  and name != "README.md"

FOR each file in candidate_files:
  content = read(file)
  bracket_notes = extract all substrings matching "【...】"

  IF bracket_notes contains any of:
    "【通过】"
    "【已通过】"
    "【我觉得问题不大，可以通过】"
    or other explicit human acceptance phrase
  THEN node_status = "已通过"

  ELSE IF bracket_notes contains any of:
    "【不通过】"
    "【未通过】"
    "【有问题】"
    "【重测】"
    or explicit human rejection / re-test phrase
  THEN node_status = "待重测"

  ELSE
    node_status = "待验收"

  record {wbs_id, file_path, node_status}
END
```

总入口生成伪代码：

```text
records = collect_status_from_active_checklists()

pending_records = records where node_status in {"待验收", "待重测"}
passed_records = records where node_status == "已通过"

rewrite(
  "DOC/CODEX_DOC/06_测试文档/05_验收清单/00-当前待验收功能总入口.md",
  only pending_records
)

rewrite(
  "DOC/CODEX_DOC/06_测试文档/05_验收清单/README.md",
  pending_records in "当前待验收清单"
  passed_records in "已通过节点索引"
)
```

通过后同步伪代码：

```text
IF node_status == "已通过":
  rename checklist file from "[待验收]" to "[已通过]" if needed
  remove this node from "00-当前待验收功能总入口.md"
  move this node to passed section in "README.md"
  update GitHub Issue body field "当前状态" = "已验收"
  update GitHub Project field "Status" = "Done"
```

禁止行为：

1. 只看文件名中的 `[待验收]` 就认定该节点仍待验收。
2. 只看总入口或 README 的旧内容，就反推节点状态。
3. 在读取到 `【通过】` 后，只更新其中一个派生面，不更新其余协作面。
4. 把“我记得这个节点好像测过了”当成状态依据。

#### 3.5.10C 用户触发词响应规则

当用户提出以下任一类请求时：

- `生成验收入口`
- `更新待验收入口`
- `更新待验收功能总入口`
- `我要验收`
- `给我验收入口`
- `整理待测试项目`

默认处置顺序必须是：

1. 先刷新或至少复核当前集中验收总入口。
2. 再确认总入口中的节点集、数量、顺序和时间戳是否已经是本轮最新状态。
3. 如有已通过节点尚未移出、已重测节点尚未刷新、说明段仍残留历史留痕，必须先修正，再给用户路径。
4. 只有在确认入口页已经完成本轮刷新后，才能把该路径发给用户作为正式验收入口。

换句话说，用户说“我要验收”时，Codex 不能直接复读旧路径，而必须先完成一轮入口刷新检查。

#### 3.5.11 测试页与测试用例可理解性规则

- 当项目使用独立测试页、局部验证页、作者工具链验证页或专用测试用例作为验收入口时，默认必须把“可理解性”视为交付要求的一部分，而不是附带优化。
- 测试页本身应尽量做到“不读源码也知道它是干什么的、为什么点这个按钮就能验证目标”。
- 因此，测试页或测试说明默认至少应明确写出：
  - 页面用途
  - 页面验证目标
  - 关键操作为什么能证明目标成立
  - 关键词或内部术语是什么意思
  - 与其他测试页、测试场景、主流程页面之间的关系
- 如果验证链路涉及跨页、刷新、重新进入运行时、浏览器存储或临时 override，必须显式区分：
  - 什么只影响当前页面内存
  - 什么会写入浏览器存储
  - 什么会影响刷新后的其他页面
  - 什么不会自动同步到已打开页面
- 不允许默认把 `override`、`pack`、`consume`、`runtime`、`probe` 之类开发侧术语直接丢给验收者而不给解释。
- 说明区的排版应优先服务顺序阅读；当说明内容较长时，默认优先纵向布局，而不是为了“信息密度”强行横向铺卡片。
- 对这类页面的自测，不只检查功能链路，也应检查说明文案与页面结构是否仍满足上述可理解性要求。
- 如果项目使用 `codex_regression_runner`、共享 smoke 页、共享 probe 页等“一个页面承载多个节点断言”的入口做人机联合验收，验收清单还必须明确写出：
  - 当前节点到底只看哪些断言
  - 同页其余失败项是否属于当前节点
  - 这些失败项由哪个节点承接、在哪里跟踪
- 不允许把“共享回归页上还有很多 FAIL”这种需要开发者解释的状态直接丢给验收者自行判断。

#### 3.5.11A 资源化功能真实消费校核规则

- 当用户明确要求“图片化 / 素材化 / 资产化 / 资源化”时，默认不得把“字段已预留”或“程序生成占位图”误报为已实现。
- 这类需求只有在以下条件同时成立时，才可视为已真正打通：
  - 数据契约中已有稳定的资源引用字段
  - 页面或运行态实际消费了真实资源路径，而不是 data-url 占位或程序即时绘制假图
  - 校验链能稳定识别缺失资源引用
  - 验收页能直接解释资源方案是什么、图片字段在哪里、为什么这就代表真实链路成立
- 对 `JSON + image assets` 一类方案，默认推荐口径为：
  - 结构 JSON 中只保存短引用
  - 真实图片路径集中维护在 `assetLibrary` 或等价资源表
  - 编辑页、预览页、IO 页共用同一资源解析规则
- 对这类页面的自测，不只检查点击链和字段存在，还应至少额外检查：
  - 是否真的渲染了目标图片路径
  - 是否仍存在文本溢出、布局压缩、不可读区域
  - 是否把“正式输出”和“调试元信息”混在同一视觉主区
- 如果用户关注的是最终可见效果，则默认至少补一轮可视化复核：
  - 浏览器截图检查
  - 或其他等价的可视证据
  - 不能只用单元测试替代布局与表现核查

#### 3.5.12 Chrome MCP / 浏览器阻塞处置规则

- Chrome MCP、DevTools MCP、浏览器自动化页都只是验证与观察手段，不是研发主链本身；不允许因为浏览器卡顿、页面挂起、导航超时而让整轮研发长期停住。
- 任何浏览器型 MCP 调用默认都应采用“限时 + 限次 + 可降级”策略：
  - 单次页面导航、打开或关键交互默认应在可感知的短时间内完成；
  - 如果一次调用已经明显卡住，不应机械连续重复同一动作；
  - 默认最多做一次同路径重试，第二次仍异常时立即转入降级路径。
- 当出现浏览器型 MCP 卡顿、无响应、页面上下文损坏或工具链中断时，默认处置顺序为：
  1. 停止继续在同一损坏页面上下文上叠加点击、等待和脚本调用；
  2. 优先关闭坏页或新开干净页面，而不是在旧页面上持续重试；
  3. 如果重开后仍异常，立即改用非浏览器验证路径继续推进；
  4. 只有当“视觉观察本身”是唯一剩余阻塞项时，才把该问题显式记录为阻塞。
- 非浏览器降级路径默认包括但不限于：
  - 直接阅读本地 HTML / JS / CSS / 文档；
  - 通过本地服务、`curl`、`fetch`、`gh`、自动化脚本、回归页、单测或 smoke 脚本获取证据；
  - 使用 DOM 级断言、数据级断言、事件级断言替代人工点击链；
  - 对本地页面优先做“代码验证 + 自动回归验证”，而不是把 Chrome MCP 当成唯一入口。
- 如果浏览器 MCP 失效但仍存在可替代验证路径，默认要求是：
  - 继续完成可完成的实现、自测、文档和 GitHub 同步；
  - 在自测报告或交接记录中单独注明“Chrome MCP 阻塞”以及已采用的替代验证方式；
  - 不把“浏览器工具卡住了”直接等价为“本轮无法继续工作”。
- 只有在以下条件同时成立时，才允许把任务状态标记为 `阻塞中`：
  - 当前剩余问题确实必须依赖浏览器视觉验证；
  - 已完成一次重开页面/上下文的恢复尝试；
  - 已明确不存在足够可靠的非浏览器替代验证路径；
  - 已在文档中写清阻塞点、影响范围和下一步需要用户提供的支持。
- 对需要频繁使用浏览器 MCP 的工作，默认还应提前设计“脱离浏览器也能跑”的最小验证基线，例如：
  - 回归页
  - probe 页
  - 数据样本断言
  - 事件链断言
  - CLI 级 smoke 检查
- 如果某类页面长期依赖 Chrome MCP 且经常卡顿，应视为验证设计问题，而不只是工具偶发故障；后续切片中应补建更稳定的自测入口，减少对浏览器人工驱动链路的单点依赖。

#### 3.5.8 无 Tracker 时的临时策略

- 如果项目尚未建立 GitHub Project 或同类工具，可暂时由本地 WBS 计划文档承担排程视图。
- 一旦项目进入多阶段交付、多人协作或并行研发状态，应尽快建立 `Tracker`。
- 建立 `Tracker` 后，应以其作为人类查看的主计划入口，本地文档继续承担长说明与证据职责。

#### 3.5.9 GitHub 协作面路径同步规则

- GitHub Issue、GitHub Project、PR 评论等共享协作面中的文件路径，默认只使用仓库相对路径。
- 如果需要可点击打开，再补充仓库 URL；不使用 `/home/...` 这类工作站绝对路径。
- 当本地文档目录、文件名或分类结构发生调整时，必须在同一轮同步修正相关 Issue、Project 和共享说明中的路径引用。
- 不允许出现“仓库目录已迁移，但 GitHub 协作面仍指向旧目录”的漂移状态。

## 4. 架构策略

### 4.1 分层顺序

推荐按以下顺序推进：

1. 共享模型
2. 接口契约
3. 后端领域逻辑
4. 实时/事件链路
5. 前端状态层
6. 页面与展示层
7. 验收与文档

### 4.2 抽象优先

- 出现第二处相似逻辑时，就应评估是否需要抽象。
- 当系统存在“可切换模式”“多实现策略”“多页面复用”时，应优先引入统一抽象层。
- 上层业务不应直接依赖底层实现细节。

### 4.3 配置、显示、规则分离

- 配置决定系统应如何运行。
- 规则决定系统如何判定。
- 显示决定系统如何呈现。
- 三者可以关联，但不能混为一体。

### 4.4 运行态优先

- 页面展示尽量读取运行态数据。
- 规则判断尽量基于正式运行态和正式契约。
- 本地预览、实验参数、调试状态必须与正式运行态区分清楚。

## 5. 改动控制策略

### 5.1 边界控制

- 不修改与本轮目标无关的模块。
- 不顺手重构无关代码。
- 不把临时便利性改动扩散为全局耦合。

### 5.2 兼容控制

- 默认优先做干净升级，不主动保留“兼容字段”“兼容分支”“双写逻辑”“旧新并存入口”。
- 只有在以下情况之一成立时，才允许引入兼容层：
  - 用户明确要求保留兼容
  - 外部依赖、存量数据或联调对象尚未完成切换
  - 当前轮若直接切断兼容会导致无法验证或无法交付
- 一旦保留兼容层，必须同时写清：
  - 为什么必须保留
  - 兼容范围是什么
  - 谁仍在消费旧口径
  - 何时移除
- 不允许把“先留兼容字段以后再说”当成默认工程习惯。
- 新字段或新模型默认直接建立唯一真值，不先人为制造第二套并存语义。
- 如果兼容层已经成为排障复杂度来源，应优先把“移除兼容层”纳入正式 WBS/Issue，而不是长期悬置。
- 不在缺少迁移路径时直接破坏现有链路；但“没有迁移路径”必须是真实外部约束，不得作为保留脏兼容的惯性借口。

### 5.3 风险控制

- 对高风险改动优先采用“新增并接入”而不是“原地大改”。
- 对真实业务配置、存量数据、用户未提交改动保持保守策略。
- 构建产物是否纳入版本控制必须单独判断，不混入业务判断。

## 6. 验证策略

### 6.1 最低验证要求

每轮至少应完成以下内容：

- 类型检查通过
- 构建通过
- 最小接口联调或关键链路验证
- 生成人工验收清单
- 用户基于验收清单给出明确验收结果

### 6.2 验证顺序

推荐顺序：

1. 静态检查
2. 构建检查
3. 接口验证
4. 页面验证
5. 验收清单复核

### 6.3 验证标准

- 不只验证“代码能运行”，还要验证“需求是否真实生效”。
- 不只验证单点页面，还要验证跨页面和跨模块一致性。
- 不只验证新增能力，还要确认没有明显回归。

### 6.4 人工验收清单策略

- 每轮有实际改动时，默认必须生成对应验收清单。
- 验收清单默认存放在：
  - `DOC/CODEX_DOC/06_测试文档/05_验收清单/`
- 对具有明确 `WBS` 节点来源的人工验收清单，命名方式默认使用：
  - `WBS编号-[状态]-YYYY-MM-DD-HHMMSS-主题-人工验收清单.md`
- 如果是目录迁移、策略升级等不直接对应单一 `WBS` 节点的辅助型验收文档，才允许使用时间戳 + 主题命名。
- 验收清单内容至少应包括：
  - 验收对象
  - 验收范围
  - 环境记录
  - 分组验收项
  - 失败判定点
  - 建议验收顺序
  - 验收记录区
  - 对应实现文件
- 如果某个节点的人工验收入口依赖独立测试页、局部验证页或作者工具链页，则除验收清单外，该页面本身也必须具备足够自解释能力，至少能让验收者直接理解：
  1. 这是个什么页面
  2. 为什么点这个按钮能验证当前目标
  3. 关键词是什么意思
  4. 它和其他测试页、测试场景、主流程页面是什么关系
- 每次新生成验收清单后，默认必须：
  1. 将验收清单相关文档单独提交
  2. 单独推送到远端
  3. 向用户反馈对应提交号
- 验收清单提交不应混入当前轮次尚未完成的源码改动。
- `00-当前待验收功能总入口.md` 这类集中导航页，只允许列出：
  - `待验收`
  - `未通过待重测`
- `已通过` 节点不得继续留在“当前待验收”入口中。
- 如果 `已通过` 节点仍需保留人工查阅价值，应放在：
  - `README`
  - 单独索引页
  - 或归档区
  而不是继续占用“当前待验收”入口。
- 当某个 `WBS` 节点被用户明确判定人工验收通过后，同一轮必须同步完成：
  1. 将对应清单文件状态更新为 `[已通过]`
  2. 从“当前待验收功能总入口”移除该节点
  3. 更新 `Execution Contract` 中的当前状态
  4. 更新 `Tracker` 中的粗粒度状态
- 这里的“用户明确判定”默认以验收清单中的 `【通过】`、`【不通过】`、`【部分通过】` 等显式批注为准，不以开发者预填字段为准。
- 如果人工验收依赖共享回归页或共享 probe 页，同一轮还必须同步检查：
  1. 验收清单是否写清当前节点的断言范围
  2. 非当前节点失败项是否已指向明确承接节点
  3. 用户是否能从文档直接知道“这些 FAIL 现在放在哪里处理”

### 6.5 闭环判定

- 代码完成、构建通过，不等于当前阶段闭环。
- 当前阶段只有在以下条件同时满足后才算闭环：
  1. 本轮改动已完成实现
  2. 已生成对应验收清单
  3. 已告知用户按清单进行人工验收
  4. 用户已明确回复验收结果
  5. 若用户提出验收意见，已形成意见处理文档并完成审定
- 如果用户尚未回复验收结果，则当前阶段状态应标记为：
  - `待人工验收`
  - 或 `待用户确认`

### 6.6 验收意见处理策略

- 用户在验收清单中的批注、总结和结论应视为正式验收输入。
- 默认只把用户写在 `【】` 内的批注视为正式用户判断口径。
- 开发者编写的 `状态`、`验收结论`、复选框或占位字段，只是待同步的派生信息，不构成用户输入本身。
- 某份活跃清单一旦已经吸收过用户批注并准备进入下一轮重测，旧批注不继续保留在活跃清单正文中。
- 默认做法是先在 `07_过程文档/03_验收意见处理/` 或归档快照中留痕，再刷新出干净版活跃清单。
- 开发者在新写的活跃清单、策略文档和说明文档中，不主动使用用户批注保留标记。
- 对用户用特殊标记显式圈定的意见，默认按高优先级处理。
- 每次收到新的验收意见后，必须新增一份对应的处理文档，而不是只在对话里口头总结。
- 处理文档默认存放在：
  - `DOC/CODEX_DOC/07_过程文档/03_验收意见处理/`
- 命名方式应保持精确时间戳 + 主题 + `验收意见处理` 的形式。
- 建议格式：
  - `YYYY-MM-DD-HHMMSS-主题验收意见处理.md`
- 文档标题或文档信息中的创建时间必须同时包含：
  - 日期
  - 时间
- 不允许只写日期、不写时间，因为同一天可能存在多轮验收意见处理。
- 处理文档至少应包括：
  - 来源验收清单
  - 用户意见摘录
  - 问题归类
  - 原因分析
  - 影响范围
  - 处理建议
  - 待用户审定项
- 在用户审定前，不直接执行具有方向性影响的处置动作。
- 一旦处理文档或处理结论发生更新，应自动完成：
  1. 单独提交相关文档
  2. 推送到远端
  3. 告知用户对应提交号

## 7. 文档与交接策略

### 7.1 文档必须回答四个问题

每次阶段文档或交接文档至少要说明：

1. 这轮目标是什么
2. 这轮已经完成了什么
3. 还有什么没完成
4. 下一轮应该从哪里继续

### 7.2 交接必须可执行

- 交接文档必须包含路径、文件、命令和建议顺序。
- 交接内容必须让下一个执行者能在短时间内恢复上下文。
- 不允许只写模糊结论，不写具体落点。

### 7.3 验收文档必须联动

- 每次新增验收清单后，交接文档应明确引用该验收清单路径。
- 如果存在“当前待验收功能总入口”这类导航页，交接文档引用时应确保其中只包含待验收或待重测节点。
- 如果本轮尚未收到用户明确验收结果，交接文档必须写明：
  - 哪份验收清单待执行
  - 当前闭环状态尚未完成
  - 下一轮应先收集用户验收反馈
- 新增验收清单一旦生成，其独立提交号也应在必要时可追溯。

### 7.4 验收意见文档必须联动

- 如果用户已经在验收清单中给出批注或结论，交接文档必须同时引用对应的验收意见处理文档。
- 验收意见处理文档与原验收清单必须成对可追溯。
- 后续计划调整、代码修改或页面修订，应明确引用其对应的验收意见处理文档。
- 审定意见处理文档在形成或更新后，默认应同步提交并推送到远端，保持本地与远端审阅上下文一致。
- 如果同一天存在多份验收意见处理文档，必须依靠标题和文件名中的时间区分，不允许使用仅日期命名。

## 9. 共享工程默认使用说明

针对后续采用本策略的工程，默认遵守以下落地规则：

- 开始新一轮开发前，优先读取：
  - `DOC/CODEX_DOC/07_过程文档/01_会话交接/`
  - 当前激活的 `04_研制计划/` 节点文档
  - 当前阶段验收清单
- 每轮开发完成后，默认在 `DOC/CODEX_DOC/06_测试文档/05_验收清单/` 下新增对应验收清单。
- 新增验收清单后，默认单独提交并推送到远端，不等待额外提醒。
- 向用户汇报完成情况时，必须同时给出本轮验收清单路径，并等待用户明确回复验收结果。
- 在用户未明确回复“通过 / 不通过 / 发现问题”之前，不将本轮标记为开发闭环。
- 如果项目存在“当前待验收功能总入口”，默认只把 `待验收` 与 `待重测` 节点放进去；`已通过` 节点应移出该入口。
- 如果用户已在验收清单中批注意见，必须先在 `DOC/CODEX_DOC/07_过程文档/03_验收意见处理/` 下新增处理文档，待用户审定后再修改计划或代码。
- 对审定意见处理文档及其策略更新，默认自动提交并推送到远端，不等待额外提醒。
