# Engineering Work Strategy

本文件用于固化跨项目复用的共享工程工作策略，作为后续开发、迭代、重构、验收、发布和交接的统一执行基线。

本版本将策略重构为 12 个母策略一级章节：

1. 本地文档治理
2. 远端协作与工程跟踪治理（Tracker）
3. WBS 与计划治理
4. 任务执行约定治理
5. 状态机与关闭门
6. 主目录与隔离工作区治理（Worktree）
7. 事实源判定与同步治理
8. 验证、验收与证据治理
9. 会话交接与多轮协同
10. 架构治理
11. 改动边界与兼容风险
12. 版本发布与对外导出

其中 `1-9` 默认采用统一骨架：

- `目标与边界`
- `事实源`
- `强制执行门`
- `同步义务`
- `例外或降级规则`
- `产物与证据`

以下 3 个章节采用专用骨架：

- `架构治理`
- `改动边界与兼容风险`
- `版本发布与对外导出`

## 1. 本地文档治理

### 1.1 目标与边界

- 本地正式文档根是默认用户审阅入口，也是后续实现、验收、交接和追溯的正式依据。
- 文档不是附属说明，而是工程协作的一部分；设计变更、边界调整、阶段结论和交接结果都必须落文档。
- 正式文档体系采用“标准兼容型本地研发文种体系”，不是对任一国际标准或军标目录的逐项照抄。
- 工作层文档，如 `docs/superpowers/`、`.superpowers/brainstorm/`，用于方案推演、中间草稿、过程留痕和局部计划拆解，不承担正式已确认基线职责。

### 1.2 事实源

- 正式文档根中的 `README.md`、`00-本地工程策略映射.md`、稳定设计文档、稳定计划文档和正式验收资产，是本地文档事实源。
- `00-本地工程策略映射.md` 必须显式写出：
  - 当前文档管理依据采用哪组标准控制点
  - 本地正式文档根与标准投影层之间的关系
  - 当前项目启用了哪些本地扩展文种，以及这些文种分别解决什么职责缺口
- 用户默认只被认为审阅了正式文档根中的内容，而不是工作层文档。

### 1.3 强制执行门

- 在以下任一动作之前，必须先把关键结论同步进正式文档根：
  - 表述“已设计 / 已明确 / 已规划 / 可以开发 / 按此执行”
  - 编码实现
  - 自测与人工验收准备
  - 编写交接文档
  - 正式 `git commit`
  - 正式 `git push`
- 新稳定文档默认采用中文主标题；如必须出现非显然英文术语、缩写或产品词，首次出现应采用中英配对。
- 稳定正文应尽量直接按语义落到正式文档根对应目录中，不允许只在索引页挂一个“工作层引用”替代同步。

### 1.4 同步义务

- 一旦正式文档目录、文件名或分类结构发生调整，必须在同一轮同步修正相关 Issue、Project、共享说明和模板中的路径引用。
- 工作层已经沉淀出会影响后续实施的关键设计、关键计划或长期约束时，必须同轮同步到正式文档根。
- 文档结论一旦成为实施依据，就不得继续只停留在草稿、过程稿或工作笔记中。

### 1.5 例外或降级规则

- 如果项目尚未建立正式文档根，应优先创建，再继续沉淀稳定结论；不能长期用工作层目录代替正式层。
- 如果某轮只能先写工作层材料，必须明确标注其身份为草稿、过程稿或内部推演材料，不得称其为正式设计、正式计划或用户已审阅结论。

### 1.6 产物与证据

- 正式文档根默认采用以下七类结构：
  - `01_需求分析/`
  - `02_设计说明/`
  - `03_规范与流程/`
  - `04_研制计划/`
  - `05_节点合同/`
  - `06_测试文档/`
  - `07_过程文档/`
- 核心根文档默认包括：
  - `README.md`
  - `00-本地工程策略映射.md`
- 典型本地扩展文种包括：
  - `05_节点合同/`
  - `06_测试文档/01_验收大纲/`
  - `06_测试文档/02_验收入口/`
  - `07_过程文档/`

## 2. 远端协作与工程跟踪治理（Tracker）

### 2.1 目标与边界

- 远端协作面承接“共享可见的工程跟踪”，而不是承接所有长篇设计、长篇验收脚本和过程细节。
- 远端协作面可能包括 GitHub Issues、GitHub Project、GitLab board 或其他等价系统，但并非“能访问到”就自动视为正式治理面。
- `Tracker` 层主要负责排程、状态、日期、路线图和树结构可见性；`Execution Contract` 层负责单任务最小执行约定；`Local Docs` 层负责长说明与证据。

### 2.2 事实源

- 正式采用的远端看板、任务单和其字段配置，是远端协作事实源。
- GitHub Issue、PR 评论等协作面中，`Collaborator`、Owner、Maintainer 等协作者评论可直接作为执行输入；非协作者评论只能作为候选意见。
- 远端协作面中的路径默认使用仓库相对路径；如需可点击打开，再补充仓库 URL。

### 2.3 强制执行门

- 在使用 GitHub Project、GitLab board 或其他可写远端协作面之前，先判断该协作面是否已被当前项目正式采用为 `Tracker` 或共享执行面。
- “能访问 / 能看到 / 技术上能创建”不等于“当前轮可以改”。
- 只要动作会改变远端协作面的结构或协作者读写方式，例如新增字段、状态项、视图、自动化或其他 `schema`，就必须先确认：
  - 当前项目是否允许使用该远端工具
  - 当前轮是否允许代理修改远端 `schema`
- 在未获得明确允许前，可以读取远端现状，但不得擅自创建、重命名、删除或重定义远端结构。
- 非协作者评论若要触发计划或代码变更，必须先获得用户明确确认。

### 2.4 同步义务

- 当本地文档目录、文件名、分类结构或执行口径变化时，必须在同一轮同步修正远端协作面中的路径、字段说明和引用关系。
- 用户明确给出通过、确认或验收结论后，必须在同一轮同步：
  - `Tracker` 状态
  - 执行约定中的 `当前状态`
  - 本地索引或状态页
- 远端协作面中不得长期保留已经失效的旧路径、旧字段解释或旧流程说明。

### 2.5 例外或降级规则

- 如果项目尚未正式采用任何远端 `Tracker`，本地计划文档可暂时承担排程视图，但一旦进入多阶段交付、多人协作或并行研发，应尽快建立正式 `Tracker`。
- 在未获写权限或 schema 变更授权时，只能读取远端信息，不得把“已读到现状”误报为“已获得修改许可”。

### 2.6 产物与证据

- `Tracker` 层推荐最小字段包括：
  - `Status`
  - `Phase`
  - `Contributes To`
  - `Work Type`
  - `WBS Level`
  - `WBS Node`
  - `Parent Node`
  - `Start Date`
  - `Target Date`
- `Execution Contract` 层默认采用 Issue、任务单或本地节点合同承接单任务执行约定。
- 协作面引用路径默认示例：
  - `DOC/CODEX_DOC/04_研制计划/01-WBS-0-<project>-研发总纲-研制计划.md`
  - `src/server/routes.js`

## 3. WBS 与计划治理

### 3.1 目标与边界

- WBS 是稳定计划和任务树的主分解轴，用于表达对象边界、阶段结构、工作包和执行任务。
- 里程碑如 `M1 / M2 / M3` 是验证门，不是顶层分解轴。
- 计划治理关心的是“对象如何拆、顺序如何排、节点如何命名”，而不是报告归档本身。

### 3.2 事实源

- GitHub `Issues + sub-issues` 是 WBS 树的唯一结构事实源。
- GitHub Project 或同类看板只是排程与展示投影，不替代 Issue 树成为结构事实源。
- 本地 `issue tree mirror` 或其他派生镜像文件，只能视为历史快照、过程证据或辅助说明。

### 3.3 强制执行门

- 在确认以下任一信息时，必须优先读取远端 WBS 事实源：
  - 当前 WBS 树结构
  - 当前节点标题
  - 当前节点正文契约
  - 当前父子挂接关系
  - 当前远端状态字段
- WBS 命名必须优先暴露对象边界，不得先写抽象动作、再把真实对象藏在正文里。
- 同层兄弟节点必须使用同一拆分轴；如果兄弟标题无法只看标题就判断对象边界是否完整，说明拆分有问题。
- 稳定计划文档按 WBS 节点命名，不使用“当前阶段计划”作为稳定文件名。
- `04_研制计划/` 默认一节点一文档，文件名统一使用 `NN-WBS-节点编号-主题-研制计划.md`。

### 3.4 同步义务

- 同层兄弟节点顺序必须在以下三处同轮保持一致：
  - sub-issue 顺序
  - issue body 子项索引
  - tracker 项目顺序
- 当前激活节点应记录在 README、索引文档和 `Tracker` 中，而不是通过改计划文件名表达。
- 对于进入执行队列的节点，应同步明确开始时间与目标时间；时间基线优先落到 `Tracker` 路线图视图。

### 3.5 例外或降级规则

- 如果远端暂时不可访问，只允许回退到仓库主工作树或用户检查目录中的正式文档根作为本地快照参考。
- 使用本地快照时，必须明确标注为：
  - `本地快照`
  - 或 `待远端复核`
- 隔离 `worktree` 中的派生镜像不得反推当前远端 WBS 事实。

### 3.6 产物与证据

- 稳定计划主文档默认放在 `04_研制计划/`。
- `README.md`、计划索引和 `Tracker` 路线图共同承接“当前激活节点 + 时间基线 + WBS 可见性”。
- 合格的 WBS 标题应能只看标题就判断：
  - 本层对象边界是否完整
  - 兄弟节点之间是否存在漏项
  - 验证页或样本页属于哪个正式对象域

## 4. 任务执行约定治理

### 4.1 目标与边界

- 每个执行任务都应有一个最小可执行约定，保证后续执行者不依赖聊天上下文也能开工。
- 该约定承接“做什么、依赖什么、产出什么、如何验收、当前在哪个状态”，不负责承接长篇背景说明和全局路线图。
- 本章覆盖本地节点合同、Issue 正文契约和其他等价执行单元，不限定唯一载体。

### 4.2 事实源

- 单任务执行约定以对应 Issue 正文、本地 `05_节点合同/` 文档或等价正式任务单为事实源。
- 对数据规范、文件规范、导入导出类任务，约定中的成果物必须点名至少一个高字段覆盖度样例。

### 4.3 强制执行门

- 每个执行任务至少应包含以下字段：
  - `工作目标`
  - `工作内容`
  - `潜在难点`
  - `Depends On`
  - `成果物`
  - `验收方法`
  - `当前状态`
  - `上级节点`
- 如需扩展，可加入：
  - `Owner`
  - `Write Scope`
  - `证据链接`
- 如果任务主要涉及数据、文件或导入导出契约，验收方法必须包含：
  - 一个有效样例
  - 一个或多个无效样例
  - round-trip 或等价端到端一致性检查
- 不得用抽象的“验收入口 / 验收标准”替代真实可执行的验证动作。

### 4.4 同步义务

- 当任务进入 `已自测` 或 `待人工验收` 时，执行约定必须回链到：
  - 机测记录
  - 验收入口
  - 会话交接或交付记录
- 当任务验收通过、被阻塞或重新返工时，执行约定中的 `当前状态` 必须与本地和远端状态语义保持一致。
- 约定中的成果物路径优先使用仓库相对路径，不写工作站绝对路径。

### 4.5 例外或降级规则

- 如果项目未启用本地节点合同文档，可由 Issue 正文或等价任务单承接执行约定，但字段完整度不降级。
- 如果某任务过小不值得单独成文，也至少要在对应执行面写全最小字段，不得只留一句“后续补充”。

### 4.6 产物与证据

- 启用本地节点合同时，默认放在：
  - `05_节点合同/NN(.NN)-节点编号-主题-节点合同.md`
- 对应实现、样例、设计说明和验收资产都应能从执行约定中直达。

## 5. 状态机与关闭门

### 5.1 目标与边界

- 本章定义执行状态语义、状态流转边界，以及“研发推进门”和“阶段关闭门”的关系。
- 目标是把“排程视图”和“执行语义”解耦，避免一个字段同时承担排程、执行、验收和关闭判定的全部责任。
- 人工验收不是验证文档本身，而是阶段关闭门的关键判定条件。

### 5.2 事实源

- 执行状态的细粒度事实源是执行约定中的 `当前状态`。
- 人工验收通过与否的正式事实源是活跃验收方法中的用户 `【】` 批注，而不是开发者预填复选框或汇总页派生字段。
- 粗粒度排程状态由正式 `Tracker` 承接，但其语义必须受本章定义约束。

### 5.3 强制执行门

- 细粒度执行状态默认包括：
  - `待开发`
  - `开发中`
  - `已自测`
  - `待人工验收`
  - `已验收`
  - `阻塞中`
- 双门模型必须分开：
  - `研发推进门`
  - `阶段关闭门`
- 用户人工验收是阶段关闭的唯一依据，但不应默认阻断低耦合的后续准备工作。
- 下游工作分为：
  - `Prep`：低耦合准备工作，如契约草案、测试脚手架、样本数据、独立壳层、验收基线、只读分析
  - `Core`：依赖上游稳定输出的正式实现
- 当上游仍处于 `待人工验收` 时：
  - `Prep` 可以继续
  - `Core` 不应视为正式主线开始
- 若 `Tracker` 使用 GitHub Project 且 `Status` 为单选字段，默认推荐粗粒度映射：
  - `Todo`
  - `In Progress`
  - `Hum Check`
  - `Done`

### 5.4 同步义务

- 当用户明确给出通过、批准、确认或接受结论后，必须在同一轮同步：
  - `Tracker` 粗状态
  - 执行约定中的 `当前状态`
  - 本地索引或节点状态页
- 当任务仅完成自测、仍待人工验证时，不得把粗状态推进为 `Done`。
- 如果项目提供 `Hum Check`，应优先使用；如无 `Hum Check`，可临时把“待人工验收”映射为 `In Progress`，但必须保留其预验收语义。

### 5.5 例外或降级规则

- 如果远端 `schema` 变更获准，默认推荐“双层状态模型”：
  - `Status` 保持粗粒度排程状态
  - 另增 `Execution Status`、`当前状态` 或等价字段承接细粒度状态
- 只有在以下情况之一成立时，才建议把细粒度六态直接放进远端唯一 `Status` 字段：
  - 用户明确要求单字段模型
  - 当前工具无法稳定支持第二个状态字段
  - 现有协作面已经按该单字段模型运作，继续沿用的迁移成本明显更低
- 若采用单字段模型，必须显式说明其代价：路线图/排程视图与执行细状态耦合，看板可读性和跨项目一致性下降。

### 5.6 产物与证据

- 活跃验收方法中的用户 `【通过】`、`【不通过】`、`【部分通过】` 等批注，是关闭门证据。
- `Tracker` 的 `Status` 与执行约定中的 `当前状态` 共同构成状态机投影。
- 关闭门通过后留下的同步结果至少应体现在：
  - 验收方法状态
  - 执行约定状态
  - 远端 `Tracker` 状态

## 6. 主目录与隔离工作区治理（Worktree）

### 6.1 目标与边界

- 该治理对象不是“是否喜欢用 worktree”，而是正式交付目录、用户检查目录与隔离开发目录之间的职责边界。
- 默认要区分三类目录：
  - `仓库主目录`
  - `用户检查目录`
  - `隔离 worktree`
- `worktree` 只承接内部开发中间态，不承担默认正式交付职责。

### 6.2 事实源

- `00-本地工程策略映射.md` 中关于仓库主目录、用户检查目录、正式文档根和 worktree 政策的定义，是目录治理事实源。
- 默认 `git status`、`git commit`、`git push` 所基于的主工作树，应与用户认可的正式交付目录一致。

### 6.3 强制执行门

- 只有在以下情况才允许建立隔离 `worktree`：
  - 用户明确要求隔离开发
  - 需要并行处理多个低共享状态任务
  - 当前任务具有较高试验性，直接在主工作树推进风险过高
- 使用 `worktree` 后，最晚必须在以下任一时刻之前同步回主目录：
  - 准备让用户打开文件检查之前
  - 准备让用户进行人工验收之前
  - 准备向用户汇报“已完成 / 已修复 / 已可测试”之前
  - 准备启动本地服务供用户验证之前
  - 准备更新正式设计、计划、验收、自测或交接文档之前
  - 准备执行正式 `git commit` / `git push` 之前
  - 本轮已形成用户可见成果且会话即将结束之前
- 默认直接在 `main` 分支推进；只有用户明确要求隔离分支，或任务确实需要高风险试验隔离时，才建立功能分支。

### 6.4 同步义务

- 面向用户的正式文件路径、页面入口路径、验收路径和文档路径，默认只允许引用：
  - 仓库主目录
  - 或用户检查目录
- 不得把 `.git/worktrees/...`、`.config/.../worktrees/...`、临时沙盒路径或代理工作目录当成正式成果路径发给用户。
- 默认按“一个可检查切片一次同步”执行，不跨多个切片长期积压。

### 6.5 例外或降级规则

- 如果改动仍只存在于隔离 `worktree`，则：
  - 不得称其为正式已交付成果
  - 不得把该路径作为用户批注入口
  - 不得据此更新共享协作面中的正式路径引用
  - 不得把隔离目录状态误报为主目录已同步状态

### 6.6 产物与证据

- 本地策略映射文档必须显式写出：
  - 仓库主目录
  - 用户检查目录
  - 正式文档根
  - 当前是否允许使用隔离 `worktree`
  - 若允许，worktree 与主目录之间的同步门槛和正式落点

## 7. 事实源判定与同步治理

### 7.1 目标与边界

- 本章解决“谁说了算”和“改完以后必须同步到哪里”这两个问题。
- 它不是业务数据架构治理，而是协作治理中的事实源判定规则。
- 各对象域可以有不同事实源，不假设一个全局唯一根源。

### 7.2 事实源

- 文档边界与稳定结论：正式文档根
- WBS 结构：GitHub `Issues + sub-issues`
- 单任务执行约定：Issue 正文、本地节点合同或等价任务单
- 粗粒度排程状态：正式 `Tracker`
- 人工验收结论：活跃验收方法中的用户 `【】` 批注
- 评论执行输入：协作者评论；非协作者评论仅为候选意见

### 7.3 强制执行门

- 在引用状态、树结构、验收结论、路径映射或执行约定之前，必须先判定当前对象域的事实源。
- 不得用派生页、汇总页、隔离 `worktree` 镜像或口头印象代替正式事实源。
- 若多个来源冲突，应优先信任对象域内定义的正式事实源，并在需要时明确说明：
  - 哪个来源是事实源
  - 哪个来源是派生视图或快照

### 7.4 同步义务

- 只要正式事实源发生变化，相关投影层和衍生层就必须在同一轮同步。
- 典型必须同轮同步的场景包括：
  - 用户验收通过后，同步 `Tracker`、执行约定和本地状态页
  - 本地文档目录迁移后，同步 Issue、Project 和共享说明中的路径
  - WBS 顺序调整后，同步 sub-issue 顺序、issue body 子项索引和 tracker 项目顺序
- 不允许让事实源与投影层长期漂移，只在聊天中口头解释。

### 7.5 例外或降级规则

- 若远端事实源暂时不可访问，可回退到主工作树中的正式文档快照，但必须明确标注为：
  - `本地快照`
  - 或 `待远端复核`
- 非协作者评论在用户未确认前，只能保持为“待确认候选输入”，不得直接进入实施。

### 7.6 产物与证据

- 可追溯同步结果至少应体现在：
  - 更新后的正式事实源
  - 对应投影层的状态或路径同步
  - 必要时的交接或处理记录
- 当需要说明“为什么以 A 为准而不是 B”时，应在交接文档、处理记录或协作面中留下明确依据。

## 8. 验证、验收与证据治理

### 8.1 目标与边界

- 验证不只检查“代码能运行”，还要检查“需求是否真实生效、跨模块是否一致、有没有明显回归”。
- 人工验收、自测和证据留存属于同一治理链条，但各自面向不同读者。
- 浏览器、MCP、截图、CLI 检查、自动化测试都只是取证手段，不是完成判定本身。
- 对已有设计稿、样稿、静态原型图、HTML 原型或已通过评审效果图的 UI / 页面开发，原型效果图是实现标准的一部分，不是可选参考。

### 8.2 事实源

- 新鲜的命令输出、运行态验证结果、可视化检查结果和正式验收方法，是验证事实源。
- 用户在验收方法中的显式批注，是人工验收事实源。
- 验收大纲、验收主入口、验收方法、机测记录、人测记录和验收结论共同构成证据链，但各自职责不同。
- 对原型驱动的页面，正式原型目录、设计稿、样稿、通过评审的效果图、原型截图、运行态截图和截图比对结论共同构成设计一致性事实源。

### 8.3 强制执行门

- 每轮至少完成以下验证链：
  - 类型检查或等价静态检查
  - 构建或启动验证
  - 最小关键链路验证
  - 对有实际改动的轮次生成或更新对应验收方法，并复核验收主入口
- 若 UI / 页面已有设计稿、样稿、静态原型图、HTML 原型或已通过评审效果图，则必须额外完成原型截图比对门：
  - 开发前写明原型来源、目标路由、所属应用、页面主对象、允许偏差和不可接受偏差
  - 开发后提供原型截图、真实运行态截图和书面差异结论
  - 页型结构、核心区块、动作入口、视觉主次和对象语义不一致时，不得声称完成
  - 单测、类型检查、构建通过、接口通过、路由 `200 OK` 或“看起来差不多”的口头判断，均不能替代截图比对
- 推荐验证顺序：
  - 静态检查
  - 构建检查
  - 接口或关键链路验证
  - 页面验证
  - 原型截图与运行态截图比对
  - 验收主入口复核
- `02_验收入口/` 根目录默认承接：
  - `README.md`
  - `00-验收主入口.md`
  - 当前活跃的 `*-验收方法.md`
  - `99_历史归档/`
- 本策略中的“验收入口”是访问层和总入口层，不是节点级执行文档。
- 当前轮真正可执行、可批注、可作为人工验收事实源的文档统一叫 `验收方法`。
- 对具有明确 WBS 来源的当前活跃方法，默认命名：
  - `<WBS编码>-<主题>-验收方法.md`
- 对不直接对应单一 WBS 节点、但仍属于当前活跃验收范围的组织 / 配置 / 文种类方法，默认命名：
  - `SETTING-<范围名>-验收方法.md`
- 当前活跃验收方法不得再以时间戳作为第一排序键；时间信息写入正文元数据。
- 验收方法不得要求用户通过“读代码确认函数存在”“手工执行 npm 命令”完成验收。
- Chrome MCP、DevTools MCP 或浏览器自动化只是验证手段，不允许成为研发主链单点阻塞。
- 对有原型的页面，浏览器自动化可以降级为可靠的手工截图或脚本截图，但不能降级为完全没有视觉证据。

### 8.4 同步义务

- 每轮有实际改动时，默认创建或更新：
  - 机测记录
  - 验收方法
  - 验收主入口
  - 必要时的验收大纲、人测记录、验收结论
  - 会话交接文档
- 原型驱动页面完成后，必须把原型截图、运行态截图、差异结论或其输出目录写入自测 / 验收 / 交接证据链。
- 验收方法或验收主入口更新后，默认需要：
  - 单独提交相关文档
  - 单独推送到远端
  - 告知用户对应提交号
- 当活跃验收方法被新一轮方法替代、或旧入口结构被废弃时，必须在同一轮：
  - 更新 `00-验收主入口.md`
  - 将旧方法或旧结构迁入 `99_历史归档/`
  - 同步修正文档、执行约定和交接中的方法路径引用
- 当收到新的验收意见后，必须新增对应处理文档，而不是只在对话里口头总结。

### 8.5 例外或降级规则

- 浏览器验证默认采用“限时 + 限次 + 可降级”：
  - 同路径最多重试一次
  - 第二次仍异常时立即转入非浏览器验证路径
- 只有当以下条件同时成立时，才允许把任务标记为 `阻塞中`：
  - 剩余问题必须依赖浏览器视觉验证
  - 已完成一次页面或上下文重开恢复尝试
  - 已明确不存在足够可靠的非浏览器替代验证路径
  - 已写清阻塞点、影响范围和需要的外部支持
- 若原型截图比对因浏览器、截图工具或服务环境异常无法完成：
  - 必须记录失败命令、失败原因和替代证据
  - 可以继续做非视觉测试和局部修复
  - 不得把该页面标记为设计一致性已通过
- 历史归档中的验收入口或验收方法可保留原时间戳命名或原结构命名；仅当前活跃方法采用节点优先命名。

### 8.6 产物与证据

- `06_测试文档/` 默认结构包括：
  - `01_验收大纲/`
  - `02_验收入口/`
  - `03_机测记录/`
  - `04_人测记录/`
  - `05_验收结论/`
- `02_验收入口/` 默认进一步包括：
  - `00-验收主入口.md`
  - 根目录 `*-验收方法.md`
  - `99_历史归档/`
- `07_过程文档/03_验收意见处理/` 默认承接验收意见处理记录。
- 如果项目启用当前人工验收入口体系，其正式位置默认在：
  - 总入口：`DOC/CODEX_DOC/06_测试文档/02_验收入口/00-验收主入口.md`
  - 节点方法：`DOC/CODEX_DOC/06_测试文档/02_验收入口/<WBS编码>-<主题>-验收方法.md`
- `04_人测记录/` 默认只在真实人工验收发生后生成；如果项目因文种齐套预建空壳，必须显式写明“当前尚无用户批注，尚未形成独立人测事实”。
- 证据链应能回溯：
  - 本轮做了什么
  - 如何验证
  - 用户如何验收
  - 当前是否真正闭环
- 对有原型的 UI / 页面，证据链还应能回溯：
  - 使用了哪份原型或效果图
  - 运行态截图在哪里
  - 差异结论是什么
  - 哪些差异被判定为可接受，哪些差异阻塞通过

## 9. 会话交接与多轮协同

### 9.1 目标与边界

- 本章保证多轮开发、多人协作和用户验收节奏滞后时，后续执行者仍能快速恢复上下文。
- 交接不是“写个总结”，而是写出下一轮可执行的恢复入口、路径、命令和建议顺序。
- 会话交接、自测、验收、意见处理和计划更新必须能互相回链。

### 9.2 事实源

- 每轮开始前应优先读取：
  - 最近一次会话交接文档
  - 当前阶段计划
  - 验收主入口
  - 当前节点验收方法
  - 最近的自测或测试报告
- 多轮协同中的当前上下文，以最新正式交接文档和活跃验收资产为事实源，而不是聊天记忆。

### 9.3 强制执行门

- 开始新一轮工作前，必须明确：
  - 本轮目标
  - 本轮非目标
  - 本轮验证方式
- 结束一轮工作后，若有实际变化，默认生成或更新：
  - 交接文档
  - 对应验收方法或机测记录
  - 必要时的处理记录
- 如果用户尚未明确回复验收结果，本轮状态必须保持在：
  - `待人工验收`
  - 或 `待用户确认`

### 9.4 同步义务

- 交接文档必须同时引用本轮验收主入口和实际执行的验收方法路径；如用户已经给出批注，也必须引用对应验收意见处理文档。
- 新增验收方法、处理文档和关键策略更新形成或更新后，默认单独提交并推送到远端，不等待额外提醒。
- 后续计划调整、代码修订和状态更新，应能从交接文档追溯到对应验收意见处理文档。

### 9.5 例外或降级规则

- 如果项目尚未建立正式 `Tracker`，本地 WBS 计划文档可暂时承担排程视图，但交接文档仍需明确当前激活节点和后续顺序。
- 若同一天存在多轮交接、验收或意见处理，文件名和标题必须依赖精确时间区分，不得只写日期。

### 9.6 产物与证据

- `07_过程文档/` 默认包括：
  - `01_会话交接/`
  - `02_历史计划/`
  - 必要时的 `03_验收意见处理/`
- 交接文档至少回答四个问题：
  - 这轮目标是什么
  - 这轮已经完成了什么
  - 还有什么没完成
  - 下一轮应该从哪里继续
- 对跨项目快速迁移本策略的工程，可复用以下执行口令：
  - 先文档，后代码
  - 先契约，后实现
  - 先抽象，后扩散
  - 小步推进，持续可运行
  - 验证闭环，先于继续开发

## 10. 架构治理

### 10.1 核心目标

- 架构治理的目标不是追求抽象本身，而是用稳定边界承接真实业务变化，减少跨模块硬编码和后期返工。
- 架构优先级应服务于“先模型与契约，再实现与页面”的总顺序。

### 10.2 分层顺序

- 推荐推进顺序：
  1. 共享模型
  2. 接口契约
  3. 后端领域逻辑
  4. 实时或事件链路
  5. 前端状态层
  6. 页面与展示层
  7. 验收与文档

### 10.3 抽象与边界

- 出现第二处相似逻辑时，就应评估是否需要抽象。
- 当系统存在可切换模式、多实现策略或多页面复用时，应优先建立统一抽象层。
- 上层业务不应直接依赖底层实现细节。

### 10.4 配置、规则与显示分离

- 配置决定系统应如何运行。
- 规则决定系统如何判定。
- 显示决定系统如何呈现。
- 三者可以关联，但不应混为一体。

### 10.5 运行态优先

- 页面展示尽量读取正式运行态数据。
- 规则判断尽量基于正式运行态和正式契约。
- 本地预览、实验参数和调试状态必须与正式运行态区分清楚。

### 10.6 架构检查点

- 只要出现以下征兆，就应重新审视架构边界：
  - 页面直接拼接业务规则
  - 临时配置开始承担正式规则语义
  - 同一业务对象在多个页面上出现不一致模型
  - 旧兼容层长期阻塞新架构落地

## 11. 改动边界与兼容风险

### 11.1 控制目标

- 该章节控制“本轮改动能动到哪里、要不要引入兼容层、风险如何切分、何时必须把临时兼容债务显式化”。
- 原则是不因为一时方便把无关模块、旧兼容或高风险重写混进当前切片。

### 11.2 边界控制

- 不修改与本轮目标无关的模块。
- 不顺手重构无关代码。
- 不把临时便利性改动扩散为全局耦合。
- 对高风险改动优先采用“新增并接入”，而不是“原地大改”。

### 11.3 兼容层准入门

- 默认优先做干净升级，不主动保留兼容字段、兼容分支、双写逻辑或旧新并存入口。
- 只有在以下情况之一成立时，才允许引入兼容层：
  - 用户明确要求保留兼容
  - 外部依赖、存量数据或联调对象尚未完成切换
  - 当前轮若直接切断兼容会导致无法验证或无法交付
- 一旦保留兼容层，必须同时写清：
  - 为什么必须保留
  - 兼容范围是什么
  - 谁仍在消费旧口径
  - 何时移除

### 11.4 风险分级与处置

- 对真实业务配置、存量数据和用户未提交改动保持保守策略。
- 构建产物是否纳入版本控制必须单独判断，不混入业务判断。
- “没有迁移路径”必须是真实外部约束，不能作为长期保留脏兼容的借口。

### 11.5 技术债与移除计划

- 如果兼容层已经成为排障复杂度来源，应优先把“移除兼容层”纳入正式 WBS、Issue 或计划，而不是长期悬置。
- 不允许把“先留兼容字段以后再说”当成默认工程习惯。

### 11.6 默认执行口径

- 小步迭代、持续可运行。
- 真实生效优先于表面完成。
- 对高风险改动先缩小切片，再决定是否需要隔离工作区或独立验证页。

## 12. 版本发布与对外导出

### 12.1 发布目标与层次

- 本策略仓存在多个发布层：
  - canonical 正本
  - internal skill 包
  - public-skill 正本
  - public-repo 导出根
  - 独立公开仓 checkout
- 发布治理的目标，是保证这些层次边界清晰、内容同步、版本号和发布边界可追溯。

### 12.2 同步链

- canonical 变更后，先同步 internal skill 参考副本。
- public 版发布前，先把 public-skill 与 public-repo 导出目录对齐。
- 如果本机存在独立公开仓 checkout，再把 public-repo 导出同步到该 checkout。
- 推荐使用现有脚本链：
  - `./scripts/sync-skill-package.sh`
  - `./scripts/build-public-repo-export.sh <skill-name>`
  - `./scripts/sync-public-repo-checkout.sh <skill-name> /path/to/public-repo-checkout`

### 12.3 发布门

- 发布前至少完成：
  - canonical 与 internal skill 漂移检查
  - public-skill 清洁度检查
  - public-repo 导出漂移检查
  - 必要的 markdown 或文档 diff 合法性检查
- 不允许带着已知导出漂移或公共包私有残留直接发版。

### 12.4 对外清洁要求

- public-skill 与 public-repo 中不应残留：
  - 工作站绝对路径
  - 私有仓路径
  - 本地安装器私有说明
  - 仅对源仓内部有意义的指纹
- 对外可见包应保持公共可分发语义，不把本地工程细节泄漏出去。

### 12.5 版本边界

- 源仓提交边界与公开仓 tag 边界是两套边界：
  - 源仓负责本地正本演进
  - 公开仓 tag 负责公共发布版本
- 公共发布默认以公开仓自身 Git tag 作为版本边界，而不是以源仓 tag 直接代替。

### 12.6 发布产物

- 源仓中的 `public-skill/<skill-name>/` 是公共包正本。
- 源仓中的 `public-repo/<skill-name>/` 是公开仓导出根目录。
- 独立公开仓应同时保持：
  - 根 README 等面向人类的公开内容
  - `<skill-name>/` 安装包目录
- 发布完成后，应能明确给出：
  - 源仓提交号
  - 公开仓提交号
  - 公开 tag
  - 若存在，额外 release 说明或 readiness 记录
